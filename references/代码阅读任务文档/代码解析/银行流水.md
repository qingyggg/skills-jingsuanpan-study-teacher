## 银行流水模块架构总览

```
api/bank/          ← HTTP 接口层（7个文件）
service/s_ea/      ← 业务逻辑（bank_save.go 等）
service/s_bank/    ← PDF 解析引擎（141个银行解析文件）
model/             ← 3张核心表 + 4张辅助表
```

---

## 一、数据模型（三层结构）

### 银行账户 `ea_bank_name`（配置层）

```
ea_bank_name（一个企业可绑定多个银行账户）
├── ComId / OrgId         企业归属
├── Name                  银行名称（如"工商银行-基本户"）
├── SubjectId / FzhsId    该账户绑定的会计科目（如1002银行存款）
├── Currency              币种（CNY/USD等外币）
├── BankNo                账号
└── 招行接口相关字段：Xybh、XyEndTime、AuthStatus 等（招行API授权）
```

### 银行流水主表 `ea_bank`（流水层）

```
ea_bank（每一笔流水对应一条）
├── BankNameId            归属银行账户
├── Period                账期（如202501）
├── Type                  1=收入（借） / 2=支出（贷）
├── Amount                交易金额（人民币）
├── WbAmount              外币金额
├── OtherName             对方账户名称（摘要判断用）
├── Serialno              流水号（去重用）
├── Pdf / PdfA5           回单PDF路径（原始 + A5格式）
├── VoucherId             关联凭证ID（0=未生成）
├── From                  来源：manual手工/import导入/in进项/out销项/fee费用
└── FromId                来源业务ID
```

### 银行流水明细 `ea_bank_item`（明细层）

```
ea_bank_item（一笔流水可拆为多条明细）
├── BankId                归属流水主表
├── Type                  收/支方向
├── TransDate             交易日期
├── Amount / WbAmount     金额
├── SubjectId / FzhsId    补充的对方科目
├── BusinessTypeId        业务类型
├── Summary               摘要
├── ParentID              父级明细ID（分拆用）
└── Balance               余额（从回单解析来）
```

**两个特殊关系**：
- `ea_bank` 与 `ea_bank_item` 是主从关系，一笔流水可拆分成多条明细（例如一笔付款对应多个费用科目）
- `ea_bank_item.ParentID` 用于记录拆分前的父明细，清除补充时可以合并回去

## 四、保存核心流程（`service/s_ea/bank_save.go`）

```
SaveBank(bank, session, txItem, txMain)
  ├── 结账状态校验
  ├── 参数校验（银行账户ID、来源、明细非空）
  │
  ├── 【流水号去重】
  │     同账户+账期+流水号+金额 查一遍旧流水号、新流水号
  │     两者都不存在才继续，否则直接跳过（幂等设计）
  │
  ├── 【科目记忆自动填充】
  │     ea_bank_memory 按"账户+对方名称+收支类型"查历史科目
  │     历史有则自动补 SubjectId/FzhsId/BusinessTypeId
  │
  ├── 新增：INSERT ea_bank + ea_bank_item
  │
  └── 修改：同样更新两张表
```

## 五、PDF 解析引擎（`service/s_bank/`）

这是整个银行模块最庞大的部分，**141 个文件**，覆盖 100+ 家银行。

### 技术原理

```
PDF 文件
  → pdfcpu 库提取文本内容
    → SplitPDF() 按关键词把多页回单拆成单张
      → 各银行专属正则匹配解析
        → []EaBank（结构化数据）
```

核心工具（`service/s_bank/common.go`）：
- `SplitPDF`：按分割关键词（如「重要提示」）把一个多回单 PDF 拆成单张
- `SplitConfig`：控制拆法（按关键词/按百分比位置/按页数）
- `ReadPdf`：提取 PDF 文本内容

每家银行的解析逻辑独立成一个文件，格式类似：
```go
// gongshangBankPdf.go（工商银行）
func GsBankFile(info, comId, bankNameId, ...) ([]model.EaBank, error)
```
内部用 `regexp.MustCompile` 匹配各银行回单的固定格式，提取日期、金额、对方户名、摘要、余额等。

### 另一条路：AI 识别（`llm_bank.go`）

对于不能用正则稳定匹配的银行（扫描版或格式多变），通过 `RcpUrl = "http://8.140.254.159:8100/transformQueue"` 对接**第三方 LLM 解析服务**，返回结构化数据后同样走 `SaveBankDate` 入库。

## 六、银行余额计算（`handleBankBalance`）

```
某银行账户当前账期余额 =
    第一账期科目余额表期初值
  + Σ 各账期收入金额
  - Σ 各账期支出金额
```
从建账第一期开始逐期累加，不依赖回单上的余额字段，保证与账务数据一致。

## 八、特殊设计点

**1. 一笔流水可拆分多明细**  
`ea_bank_item.ParentID` 记录拆分关系，清除时通过父级 ID 把拆出去的金额合并还原。

**2. 招行 CMBC 单独对接**  
`api/bank/cmbc.go` + `service/s_bank/cmbc.go` 走招行官方 API 直接拉取流水，不依赖 PDF 上传。

**3. 科目记忆**  
`ea_bank_memory` 是核心体验功能：首次导入需手工补充科目，系统记住"对方名称=某科目"，下次相同来源自动填充，减少重复劳动。

**4. 去重机制**  
`Serialno`（流水号）+ `BankNameId` + `Amount` 三者联合判断，防止同一回单重复上传。外币流水用 `WbAmount` 而不是 `Amount` 比较。
