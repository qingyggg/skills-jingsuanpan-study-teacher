#### 已掌握的核心概念

1. **三步建账流程**
   - 第一步：导入和整理科目余额表（写入 `init_*` 临时表）
   - 第二步：补全标准科目体系（自动补全缺失的标准科目）
   - 第三步：正式建账（将临时数据写入 `ea_*` 正式账套表）

2. **临时表 vs 正式表**
   - `init_*` 表：建账工作台，可反复修正
   - `ea_*` 表：正式账套，面向日常业务

3. **科目编码体系**
   - AccountSystem: 会计准则类型（ComKj1-9）
   - 科目编码参数：num（层级规则）、sep（分隔符）、sepName（名称分隔符）、codeType（会计制度）
   - 科目层级关系：父子科目、兄弟科目
   - 科目排序：使用 GetSubjectCodeOrderSQL() 实现层级排序

##### 2. SubjectFormat() ✅
**文件位置**: [service/s_createBook/subject.go:13-124](../service/s_createBook/subject.go#L13-L124)

**参数说明**:
- `num`: 层级规则，如 "4-2-2-2" 表示一级4位，二级2位，三级2位

##### 3. FindBuildNewCode() ✅
**文件位置**: [service/s_createBook/buildSubject.go:256-292](../service/s_createBook/buildSubject.go#L256-L292)

**功能**: 为新增的下级科目生成科目编码

**关键点**:
- 查找父科目和兄弟科目
- 如果有兄弟科目：从最大编码递增
- 如果无兄弟科目：使用父编码 + ".01"
- 如果无父科目：返回错误

**核心逻辑**:
```go
// 判断兄弟科目的4个条件：
1. 相同等级（层级数相同）
2. 不是一级科目
3. 父科目相同
4. 兄弟科目已配置
```

**为什么要去掉点号转int比较**:
- 避免字符串比较错误："1001.9" > "1001.10" (字符串比较)
- 转为整数后：10019 < 100110 (正确的数值比较)

##### 4. FindBuildSubjectAll() ✅
**文件位置**: [service/s_createBook/buildSubject.go:236-253](../service/s_createBook/buildSubject.go#L236-L253)

**功能**: 将导入的科目匹配到系统标准科目

**三种匹配策略**:

1. **特殊科目匹配** - 生产性生物资产
   ```go
   if strings.Contains(name, "生产性生物资产") {
       // 精确匹配编码和名称
   }
   ```
   - 原因：农业企业有"生产性"和"消耗性"生物资产，会计处理完全不同

2. **损益类科目匹配** - 4/5/6开头
   ```go
   if code[0:1] == "4" || code[0:1] == "5" || code[0:1] == "6" {
       // 比较后3位 + 名称
   }
   ```
   - 原因：不同会计准则对损益类科目编码规则不同
   - 例子：外部 `4001` 可匹配标准 `6001`（都是"主营业务收入"）

3. **标准匹配**
   ```go
   if code == subjectAll.Code && name == subjectAll.Name {
       return subjectAll
   }
   ```
   - 编码和名称必须完全一致

##### 5. BuildSubjectExist() ✅
**文件位置**: [service/s_createBook/buildSubject.go:295-302](../service/s_createBook/buildSubject.go#L295-L302)

**功能**: 检查科目是否已存在于初始化暂存表中

**使用场景**: 在第二步建账中，用于判断哪些标准科目需要补全

**为什么只检查编码不检查名称**:
- 科目编码是唯一标识
- 名称可能有细微差异，但编码相同即视为同一科目

#### 核心状态字段

在 `init_company` 表中有三组状态字段：

1. **建账状态 (JzStatus)**
   - `0`: 未建账
   - `1`: 建账成功
   - `2`: 建账失败
   - `jz_err`: 建账错误信息

2. **补全状态 (BqStatus)**
   - `0`: 未补全
   - `1`: 补全成功
   - `2`: 补全失败
   - `bq_err`: 补全错误信息

3. **交易状态 (JyStatus)**
   - 用于历史凭证导入等交易操作
   - `jy_err`: 交易错误信息

```go
if err != nil {
    // 建账失败
    jz_status = 2
    jz_err = err.Error()
} else {
    // 建账成功，但需要验证损益类科目是否有余额
```

**为什么要检查损益类科目余额**:
- 损益类科目（收入、成本、费用）理论上在期初应该是0
- 如果有余额，说明可能存在数据问题

#### 状态依赖关系

**自动建账流程** - Automatic()
```
1. 先执行 AutomaticBq() 自动补全科目
   ↓ 失败则设置 bq_status = 2
   
2. 再执行 AutomaticJz() 自动建账
   ↓ 检查 BqStatus != 1 则报错"请先补全科目"
   ↓ 调用 SubjectInitImportTwo() 第二步
   ↓ 调用 SubjectInitImportThree() 第三步
   
3. 最后调用 JzStatusSave() 保存建账状态
```

**迁移流程状态检查** - move.go
```go
if info.BqStatus != 1 {
    err = errors.New(info.Name + "科目未补全")
```

#### 关键发现

1. **状态是渐进式的**
   - 必须先补全科目（BqStatus = 1）
   - 才能建账（JzStatus = 1）
   - 才能进行后续操作（如迁移、历史凭证导入）

2. **错误信息的两种形式**
   - 失败错误：`jz_status = 2, jz_err = 错误原因`
   - 成功警告：`jz_status = 1, jz_err = 警告信息`（如损益类有余额）

3. **状态重置**
   - 重新发起任务时会重置状态为 0
   - 例如：`service/s_task/oldInit.go:36` 重置 `bq_status = 0`

#### 为什么要用 MQ 异步处理

**核心原因**:
1. **数据量大**: 历史凭证可能有数千上万条，同步处理会超时
2. **计算复杂**: 需要科目映射、余额计算、上下级校验等多个步骤
3. **解耦操作**: 前端发起任务后立即返回，后台慢慢处理
4. **任务追踪**: 通过任务系统可以查看进度和结果

```
   - 调用 SaveInit() 重新计算余额

   - 调用 LinkAge() 确保父子科目余额一致

   - 成功: task_status = 2, err_log = 警告信息
   - 失败: task_status = 3, err_log = 错误信息
```

#### 为什么先落预览表

**预览表的作用**:

1. **init_voucher_pre / init_voucher_item_pre**
   - 存储从外部系统采集的原始凭证数据
   - 保留原始科目编码（old_code）
   - 可以反复修正和调整
   - 不影响正式账套

2. **正式表 init_voucher / init_voucher_item**
   - 存储经过科目映射后的标准凭证
   - 使用新科目编码（new_code）
   - 关联到 init_subject 表
   - 用于生成科目余额表

**数据流转**:
```
外部系统
  ↓ 采集
init_voucher_pre（预览表，原始编码）
  ↓ MQ 异步处理
  ↓ 科目编码映射（oldToNew）
init_voucher（正式表，新编码）
  ↓ 汇总计算
init_subject_balance（科目余额表）
```

#### 关键函数解析

##### oldToNew() - 科目编码映射
**文件位置**: [service/s_init/createHis.go:558-610](../service/s_init/createHis.go#L558-L610)

**映射逻辑**:
```go
1. 在 init_subject_old_new 表中查找 old_code
2. 获取对应的 new_code
3. 在 init_subject 表中查找 new_code 对应的 subject_id
4. 检查是否是最下级科目
   - 如果有下级科目，递归找到最下级
   - 因为凭证只能使用最下级科目
5. 返回 subject_id 和 new_code
```

**为什么要找最下级科目**:
- 会计准则规定：只有明细科目（最下级）才能记账
- 如果映射到了上级科目，需要自动找到它的某个下级科目

##### CheckXXX() - 发生额校验
**文件位置**: [service/s_init/createHis.go:407-423](../service/s_init/createHis.go#L407-L423)

**校验逻辑**:
```sql
-- 科目余额表的发生额汇总
SELECT SUM(period_int), SUM(period_out) 
FROM init_subject_balance 
WHERE period = ?

-- 凭证分录的发生额汇总
SELECT SUM(in_amount), SUM(out_amount) 
FROM init_voucher_item 
WHERE period = ?

-- 两者必须相等，否则数据有问题
```

##### CheckSubEqSup() - 上下级余额校验
**文件位置**: [service/s_init/createHis.go:425-476](../service/s_init/createHis.go#L425-L476)

**校验逻辑**:
```
1. 查询所有科目余额
2. 按 parent_id 分组，找出每个科目的下级科目
3. 对于有下级的科目：
   - 汇总所有下级的余额
   - 与上级科目的余额对比
   - 如果不相等，记录错误
4. 返回所有不相等的科目编码
```

**为什么要校验上下级**:
- 上级科目余额 = 所有下级科目余额之和
- 这是会计基本原则
- 如果不相等，说明数据有问题

```
  ↓ 第一步：导入整理
ea_subject_balance_init（临时表）
  ↓ 第二步：补全科目
ea_subject_balance_init（完整）
  ↓ 第三步：正式建账
init_subject（正式初始化表）
  ↓ MQ 异步处理
init_voucher（科目映射后）
  ↓ 汇总计算
init_subject_balance（余额表）
  ↓ 最终迁移
ea_*（正式账套表）
```

### 易错点总结

1. **科目编码不规范**
   - 外部软件的编码格式不统一
   - 解决：SubjectFormat() 统一格式化

2. **科目体系不完整**
   - 只导入旧科目，缺少标准科目
   - 解决：第二步自动补全

3. **科目映射失败**
   - 旧编码找不到对应的新编码
   - 解决：确保 init_subject_old_new 表完整

4. **上下级余额不等**
   - 数据采集或计算错误
   - 解决：CheckSubEqSup() 校验并报错

5. **损益类科目有余额**
   - 期初损益类科目理论上应该为0
   - 解决：给出警告但不阻止建账

## 第5阶段总结

### 核心收获

1. **凭证是会计中枢**
   - 发票、银行、工资、费用等业务数据，进入会计体系的标志通常就是生成凭证
   - 很多“不能修改/不能删除”的限制，本质是在保护凭证链路一致性

2. **凭证模块有三条主线**
   - 手工凭证保存：解决人工录入、修改、删除、审核
   - 一键生成凭证：解决业务单据到会计分录的自动转换
   - 预览和影像：解决正式落账前的展示校验，以及后续打印归档

3. **预览是正式落账前的模拟层**
   - 可直接看已生成凭证
   - 也可按来源类型临时拼装会计结果
   - 这是业务单据和正式账务之间的重要缓冲层

4. **自动生成凭证依赖统一口径**
   - 不同来源数据有各自的摘要、科目、税额和合并规则
   - 如果规则不统一，最终同类业务会生成不一致的分录

5. **维护凭证问题不能只看凭证本身**
   - 很多问题真正要追的是来源单据状态、科目配置、审核状态、账期限制和影像链路

**已经读到的关键结论**:
- `SaveInInvoice` 不是简单保存，而是“先校验业务成立，再决定新增/修改，再重建明细，再维护未认证状态”
- 修改进项发票时，如果已经生成凭证，会先删除旧凭证，再重建发票数据
- 明细保存前会过滤清单汇总行、校验金额税额关系、补税收分类编码、规范项目名称
- `BusinessTypeID` 来自 `ea_business_type` 主数据表，进项只使用其中 `type = 'in'` 的业务类型
- 代码里明确出现的常见进项业务类型 ID：`1` 原材料、`2` 库存商品、`5` 周转材料、`8` 工程物资、`18/19` 成本类分支
- `SettlementStatus` 不是手工随便填，而是根据往来单位绑定的会计科目类型推断是否已结算
- 红字发票在张数统计和认证状态上都走特殊规则，非采集红字票不允许随便改认证状态
- `UncertifiedInvoice()` 的职责是保证主表认证状态与未认证池记录保持一致：该入池时入池，该出池时出池
- `PeriodZz` 表示未认证发票转入认证的账期；一旦 `PeriodZz` 或 `VoucherId` 已有值，说明这张票已经进入后续链路，不能再随便修改未认证状态
- 跨账期重复票如果原来未认证、当前又观察到不再未认证，不会直接改主表，而是通过 `bjrzzt` 给未认证池记录打上“后续已发生认证变化”的标记
