# 鲸算盘财税系统功能模块与项目手册

## 1. 文档定位

这份文档用于替代 `代码阅读任务文档` 中“按阶段拆散、信息重复、模块边界不清”的阅读体验，统一形成一份更适合两类人使用的项目手册：

- 新人：先快速建立系统地图，再借助 AI 分模块学习，不一上来陷入细节。
- 老员工：按模块查入口、查主链路、查状态、查联动影响，作为日常维护索引。

这份手册不再按“第几阶段”组织，而是按“系统真实功能模块 + 核心业务主线 + 维护方法”组织。

## 2. 系统一句话概括

`etax-go-v4` 是一个面向代账机构的财税作业平台，围绕“企业账套”提供建账、单据采集、凭证生成、月末结账、报表账簿、税务申报和自动化流水线能力。

如果只记一句话，可以记成：

**这个系统的主线是：机构管理企业账套，企业在账期内完成采集、记账、结账、报税与自动化执行。**

## 3. 先看全局地图

### 3.1 系统主入口

- 启动入口：`main.go`
- 总路由入口：`router/router.go`
- 主要业务路由：`router/ea/`
- 税务任务路由：`router/tax/`

### 3.2 真实代码分层

这个项目不能只按理想分层理解，真实情况更接近：

`router -> middleware -> api/controller -> service/services -> model -> DB/MQ/RPA/OSS/Redis`

阅读时要接受几个现实：

- 模块名仍然大量使用 `etax-go-v3`
- `api/` 与 `controller/` 并存
- `service/` 与 `services/` 并存
- 同一个业务常常既有同步接口，也有异步任务和定时补偿

### 3.3 关键上下文对象

几乎所有财税业务都围绕下面 5 个对象展开：

- `OrgId`：代账机构边界
- `ComId`：企业边界
- `Period`：账期边界
- `gs_main`：税表业务对象
- `tax_task_main`：税务执行对象

新人阅读代码时，先问自己当前问题落在哪个对象上，效率会高很多。

## 4. 系统功能模块分类

下面这部分是本项目最推荐的模块分类方式。它不是按教材章节拆分，而是按真实业务闭环归并。

### 4.1 平台与组织层

这部分负责“谁在用系统、可操作哪些企业、上下文怎么切换”。

#### 1. 用户、登录与权限

- 作用：处理登录、短信验证码、用户信息、可信设备、权限边界
- 主要目录：`api/api_user`、`api/api_sys`、`api/sys`
- 关键问题：
  - 用户是否登录
  - 当前会话注入了哪些上下文
  - 可信设备、短信、密码流程是否正常

#### 2. 机构与企业账套管理

- 作用：管理代账机构、企业、账套、办税人、企业状态、附件与归属关系
- 主要路由：`router/ea/company.go`、`router/ea/org.go`
- 主要目录：`api/api_company`、`api/api_org`、`service/s_company`、`service/s_org`
- 关键对象：
  - `Org`
  - `EaCompany`
  - `ComId + Period`
  - `keep_status`
  - `device_status`
- 典型联动：
  - 办税人设备状态变化
  - 企业授权状态刷新
  - 企业归属校验

这是整个系统的入口层，后面发票、凭证、报税、自动化几乎都建立在这层之上。

### 4.2 财务底座层

这部分负责“一个企业如何拥有可记账的账套基础”。

#### 3. 建账初始化与历史迁移

- 作用：新企业建账、导入旧系统数据、生成标准科目体系、导入期初与历史凭证
- 主要路由：`router/ea/buildSubject.go`、`router/ea/init.go`、`router/ea/import.go`
- 主要目录：`api/api_init`、`api/buildSubject`、`service/s_build`、`service/s_init`、`service/s_createBook`、`service/s_import`
- 关键理解：
  - `init_*` 是建账工作台
  - `ea_*` 才是正式账套
  - 建账通常分“三步整理 + 正式落库”
  - 历史凭证导入依赖“旧编码 -> 新编码”映射

#### 4. 科目、辅助核算与基础主数据

- 作用：维护会计科目、辅助核算维度、往来单位、记忆规则等基础数据
- 主要路由：`router/ea/subject.go`、`router/ea/coop.go`、`router/ea/memory.go`
- 主要目录：`api/subject`、`api/corp`、`api/fzhs`、`api/memory`、`service/s_ea`
- 典型内容：
  - 科目树维护
  - 科目余额
  - 辅助核算维度
  - 往来单位档案
  - 记忆规则与自动匹配基础

这层是后续凭证生成和报表取数的口径基础。

### 4.3 原始单据输入层

这部分负责“业务原始事实如何进入系统”。

#### 5. 发票管理

- 作用：处理进项、销项、认证、采集、结算与入账前状态控制
- 主要路由：`router/ea/invoice.go`、`router/ea/collection.go`
- 主要目录：`api/invoice`、`service/invoice`、`service/s_ea`
- 核心特点：
  - 进项比销项更复杂
  - 发票既影响税表，又影响凭证
  - 发票采集与税务任务体系强相关

#### 6. 银行流水与资金识别

- 作用：管理银行账户、流水导入、回单识别、流水拆分与凭证生成前处理
- 主要路由：`router/ea/bank.go`、`router/ea/cash.go`
- 主要目录：`api/bank`、`api/cash`、`service/s_bank`、`service/s_ea`
- 核心特点：
  - 来源复杂，既有导入也有解析
  - 强依赖去重与摘要识别
  - 一笔流水可能拆成多条业务明细

#### 7. 工资薪酬

- 作用：管理员工档案、工资表、个税、社保公积金、工资结转
- 主要路由：`router/ea/salary.go`
- 主要目录：`api/salary`、`service/s_salary_hz`、`service/s_ea`
- 核心特点：
  - 规则计算重
  - 个税口径敏感
  - 是三大单据源中规则最强的一块

#### 8. 费用、票据、资产、存货等扩展原始业务

- 费用报销：`router/ea/fee.go`、`api/fee`
- 票据管理：`router/ea/bill.go`、`api/bill`
- 固定资产：`router/ea/asset.go`、`api/asset`
- 存货管理：`router/ea/inventory.go`、`api/inventory`、`service/inventory`

这部分不是“外围小功能”，而是凭证自动化和月末处理的重要原料层。

### 4.4 会计核算主线层

这部分是系统真正的账务核心。

#### 9. 凭证中心

- 作用：手工凭证、自动凭证、凭证模板、复制、审核、预览、打印
- 主要路由：`router/ea/voucher.go`
- 主要目录：`api/voucher`、`api/api_voucher`、`service/s_ea`
- 系统地位：
  - 凭证是大部分业务进入会计结果的边界
  - 原单是否能删改，经常取决于是否已生成凭证
  - 发票、银行、工资、费用、资产、存货最终都可能汇聚到凭证

如果把系统看成一条财务流水线，凭证就是中枢。

#### 10. 结转、结账与年结

- 作用：损益结转、资产结转、工资结转、结账、反结账、年末结转
- 主要路由：`router/ea/carryover.go`、`router/ea/year.go`
- 主要目录：`api/carryover`、`api/year`、`service/s_ea`、`service/s_year`
- 核心理解：
  - 结转是补齐期末结果
  - 结账是确认并锁定账期
  - 反结账是高风险回退动作

#### 11. 财务报表与账簿输出

- 作用：形成报表结果、账簿查询、导出打印与电子账簿输出
- 主要路由：`router/ea/book.go`、`router/ea/outPut.go`
- 主要目录：`api/book`、`api/outPut`、`api/kj`、`service/s_outPut`
- 输出类型：
  - 总账、明细账、多栏账、辅助核算账
  - 资产负债表、利润表、现金流量表
  - 凭证、账簿、报表导出与打印

### 4.5 税务作业层

这部分负责“账做完后，税怎么算、税怎么报、结果怎么追踪”。

#### 12. 算税与税表对象

- 作用：根据财务数据生成税表主数据
- 主要路由：`router/ea/taxCalculation.go`
- 主要目录：`api/taxCalculation`、`model/gs`、`service/s_tax`
- 核心对象：`gs_main`

#### 13. 税务申报任务平台

- 作用：生成申报任务、执行状态追踪、接收结果回写
- 主要路由：`router/tax/tax.go`
- 主要目录：`api/tax`、`api/notify`、`service/s_task`、`service/s_sendTax`、`service/s_taskOld`
- 核心对象：`tax_task_main`
- 核心理解：
  - 本系统负责组织任务和状态追踪
  - 外部 RPA 负责真正操作税局
  - 税务申报不是普通接口，而是任务调度系统

#### 14. 税务扩展域

- 个人所得税：`service/s_tax`、`service/s_ea`
- 企业所得税年报：`router/ea/gsnb.go`、`router/ea/year.go`、`service/gsnb`、`service/s_year`
- 发票采集和税务采集：`router/ea/collection.go`、`service/s_collection`

### 4.6 自动化执行层

这部分负责“把前面的功能串成自动流水线”。

#### 15. 自动化工厂

- 作用：围绕企业某账期推进采集、确认、补充、凭证、结账、申报、扣款等步骤
- 主要目录：`service/s_automatic`、`services/ss_automatic`、`mq`、`pkg/mq`、`cron`
- 核心特点：
  - 不是单功能，而是超级工作流
  - 通过 MQ 推进，而不是同步串行调用
  - 保留人工确认点，不是全黑盒自动
  - 强依赖企业配置、步骤状态、补偿机制

自动化工厂本质上是把“单据源 + 凭证 + 结账 + 报税”串成生产流水线。

## 5. 用业务闭环理解系统

如果新人不想按代码目录学，建议先按下面这条主线理解系统：

1. 机构登录系统，选择企业和账期
2. 企业完成建账，拥有正式会计底座
3. 发票、银行、工资、费用、资产、存货等原始数据进入系统
4. 原始单据补充规则和科目后，生成凭证
5. 月末执行结转、结账、报表、账簿输出
6. 根据账务结果算税，生成税表
7. 发起税务任务，由外部自动化链路执行申报/扣款
8. 自动化工厂把上述动作组合成可批量推进的流水线

只要记住这 8 步，阅读绝大多数模块时都能知道自己身处哪一层。

## 6. 新人学习路径

### 6.1 推荐学习顺序

建议新人不要按目录横扫，而按下面顺序学：

1. 先读全局入口：`main.go`、`router/router.go`
2. 再读平台层：机构、企业、账套、账期、授权状态
3. 再读建账与科目体系
4. 再读凭证中心
5. 再读三大单据源：发票、银行、工资
6. 再读结转结账、报表账簿
7. 最后读税务任务和自动化工厂

### 6.2 每学一个模块都回答 4 个问题

1. 这个模块解决什么业务问题
2. 它在整个财税闭环里处于哪一环
3. 路由/API/service/表分别在哪里
4. 它会影响哪些下游模块

### 6.3 新人第一周最应该搞懂的内容

- `OrgId`、`ComId`、`Period`
- 企业状态、设备状态、授权状态
- 建账三步和 `init_* -> ea_*`
- 凭证为什么是系统中枢
- `gs_main` 和 `tax_task_main` 的区别
- 自动化工厂为什么必须用 MQ

## 7. AI 辅助学习指南

这部分专门给“借助 AI 学项目”的新人使用，目标是不让 AI 只做代码翻译器，而是做项目导师。

### 7.1 推荐提问方式

不要直接问“这个文件干嘛的”，更推荐下面几类问题：

- “从业务角度讲讲企业与账套模块在系统里的作用，再带我看关键代码入口。”
- “按请求链路解释这个接口：路由、API、service、表、状态分别是什么。”
- “把发票模块和凭证模块的关系讲清楚，重点说哪些状态会互相影响。”
- “把 `gs_main` 和 `tax_task_main` 的区别讲成新人能懂的话，再给我关键目录。”
- “我在看 `service/s_ea/invoice_in.go`，请先说业务背景，再说关键函数和上下游影响。”

### 7.2 建议让 AI 固定输出的 5 项内容

每次让 AI 讲模块时，都要求它固定输出：

1. 业务意义
2. 在系统全局中的位置
3. 关键入口文件
4. 核心数据对象和状态
5. 常见问题与排查思路

### 7.3 AI 学习时的注意事项

- 不要一开始就让 AI 解释全部字段，先建立模块边界
- 不要只看 service，必须一起看路由和 API 入口
- 不要把“看懂某个函数”误认为“看懂整个模块”
- 遇到异步逻辑时，要求 AI 明确同步链路和异步链路
- 遇到状态字段时，要求 AI 讲“谁写入、谁依赖、谁展示”

## 8. 老员工维护手册

### 8.1 维护时的标准排查顺序

收到需求、bug 或线上问题时，建议固定按下面顺序排查：

1. 先定主对象：机构、企业、账期、单据、凭证、税表还是任务
2. 再找入口：路由在哪，走哪个 API
3. 再找上下文：`Token`、`OrgId`、`ComId`、`Period`
4. 再追 service 链路：是否经过 `service/`、`services/`、MQ、cron
5. 再确认落库点：改了哪张表，改了哪个状态
6. 最后看外部依赖：Redis、OSS、RabbitMQ、RPA、回调

### 8.2 高频问题分类

#### 1. 权限与归属问题

优先检查：

- `OrgId`
- 企业归属校验
- 批量操作权限校验

#### 2. 账期状态问题

优先检查：

- 当前 `Period`
- 是否已结账
- 反结账是否影响了当前修改

#### 3. 原单与凭证联动问题

优先检查：

- 是否已生成凭证
- `voucher_id`
- `from_type`

#### 4. 税务任务卡住问题

优先检查：

- `gs_main` 是否正确
- `tax_task_main` 当前状态
- 是否发出 MQ
- 外部回调是否回写

#### 5. 自动化不推进问题

优先检查：

- 自动化主记录和步骤状态
- MQ 分桶是否正常
- 企业自动化配置是否允许当前步骤
- 是否卡在人工确认点

#### 6. 建账与迁移问题

优先检查：

- `init_*` 临时表是否完整
- 旧科目映射是否建立
- 第三步正式建账是否真正完成

### 8.3 最值得长期维护的 4 份个人索引

- 高频接口索引：路由、API、service、核心表
- 核心表索引：企业、凭证、发票、银行、工资、`gs_main`、`tax_task_main`
- 状态字段索引：企业状态、授权状态、设备状态、结账状态、任务状态、自动化步骤状态
- 异步链路索引：税务任务、自动化工厂、历史导入、导出打印
